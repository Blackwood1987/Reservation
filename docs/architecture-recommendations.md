# 구조 개선 권장안

## 목적
- 현재 시스템은 시험운영용으로는 충분히 동작하지만, 기능이 계속 추가되면서 유지보수 부담이 커지고 있다.
- 본 문서는 향후 IT 부서 이관 또는 정식 운영 전환을 대비해 우선순위 중심의 구조 개선 방향을 정리한다.

## 현재 구조 요약
- 프론트엔드: `app.html` + `app.js` + `styles.css`
- 로그인: `index.html` + `login.js`
- 백엔드: Firebase Authentication + Cloud Firestore
- 배포: Vercel 정적 배포
- 권한: Firestore Rules 기반 역할 제어

## 핵심 이슈

### 1. 단일 파일 집중
- `app.js`에 인증 연동, 상태 관리, 예약 로직, 대시보드, 관리자 기능, 이벤트 바인딩이 모두 모여 있다.
- 기능 수정 시 연쇄 영향 범위가 커서 회귀 가능성이 높다.

### 2. 화면/상태/데이터 결합도 높음
- UI 렌더링 함수와 Firestore 저장/조회 로직이 밀접하게 섞여 있다.
- 특정 기능만 분리 테스트하거나 일부만 교체하기 어렵다.

### 3. 검증 체계 부족
- 현재는 `node --check app.js` 수준의 문법 검증만 존재한다.
- 예약 중복, 권한, 타임라인 정렬, 드래그/리사이즈 같은 핵심 동작을 자동으로 검증하지 못한다.

### 4. Firestore 공개 읽기 범위 재검토 필요
- `bookings`, `config` 문서가 읽기 공개 상태다.
- 게스트/작업자 조회 UX 때문에 열어둔 구조지만, 정식 운영 전에는 최소 권한 원칙으로 재조정해야 한다.

## 권장 구조 방향

### 1단계: 프론트엔드 모듈 분리
- 목표: `app.js`를 기능 단위로 분리해 수정 범위를 줄인다.
- 권장 분리안:
  - `firebase-client.js`: Firebase 초기화/Auth/Firestore 공통
  - `state.js`: 전역 상태, 선택 상태, 뷰 상태
  - `dashboard.js`: 룸맵, 타임라인, 상세 패널
  - `schedule.js`: 예약표, 드래그, 리사이즈
  - `admin.js`: 사용자/장비/Site/Room/목적 관리
  - `booking-service.js`: 예약 생성/수정/삭제/검증
  - `utils.js`: 시간/정렬/포맷/공통 헬퍼
- 기대효과:
  - 기능별 영향 범위 축소
  - 디버깅과 코드 리뷰 용이
  - 후속 인수인계 효율 상승

### 2단계: 데이터 접근 계층 분리
- 목표: 렌더링 코드와 Firestore 코드를 분리한다.
- 권장 방식:
  - Firestore CRUD를 별도 서비스 함수로 이동
  - 화면은 서비스 함수 결과만 소비
  - Firestore 문서 구조 변경 시 UI 수정 범위를 최소화
- 기대효과:
  - 백엔드 교체 또는 확장 시 대응이 쉬워짐
  - IT 부서 이관 시 서버/API 계층으로 치환하기 쉬움

### 3단계: 규칙 및 운영 정책 정비
- 목표: 시험운영용 설정과 정식운영용 설정을 분리한다.
- 권장 항목:
  - Firestore Rules 최소 권한 재설계
  - 관리자/감독자/작업자/게스트 권한표 문서화
  - 예약 삭제/반려/유지보수 등록 로그의 영구 저장 정책 수립
- 기대효과:
  - 보안 리스크 감소
  - 감사 대응성 향상

### 4단계: 테스트 최소 세트 도입
- 목표: 핵심 회귀를 자동 검출한다.
- 우선 테스트 대상:
  - 예약 중복 차단
  - 운영시간(09:00~18:00) 제한
  - 권한별 버튼/기능 노출
  - 타임라인 정렬 규칙
  - CRF 상단 고정 규칙
  - 목적별 장비 노출 제한
- 권장 방식:
  - 순수 함수 단위 테스트부터 시작
  - UI 상호작용은 소규모 시나리오 테스트로 확대

## 우선순위

### 즉시 권장
- `app.js` 모듈 분리 착수
- Firestore 읽기 정책 재정의 초안 작성
- 핵심 예약 검증 로직 함수화

### 시험운영 중 권장
- 관리자 로그/백업 절차 정식 문서화
- 장애 대응 체크리스트 정리
- 수동 QA 시나리오 문서 작성

### 이관 직전 권장
- 테스트 도입
- 백엔드/API 분리 설계서 작성
- 데이터 마이그레이션 절차 정리

## 권장 실행 순서
1. `app.js`를 기능별 모듈로 분리
2. Firestore 접근 함수를 별도 계층으로 이동
3. 예약/권한 핵심 로직 테스트 도입
4. Firestore Rules 운영 기준 재설계
5. IT 이관용 API/데이터 구조 문서 작성

## 결론
- 현재 구조는 빠른 시험운영에는 적합하다.
- 다만 장기 운영과 인수인계를 고려하면, 가장 먼저 해결해야 할 것은 `app.js` 단일화 문제다.
- 구조 분리, 권한 정비, 최소 테스트 도입 이 세 가지가 다음 단계의 핵심이다.
