# 오프라인/네트워크 불안정 대응 계획 (코드 적용 전 문서)

## 1) 목표
- 예약 데이터 유실 방지
- 중복 등록/중복 삭제 방지
- 사용자에게 현재 상태(정상/지연/오프라인) 명확히 안내
- 복구 시 자동 동기화와 충돌 최소화

## 2) 운영 상태 정의
- `정상(Online)`: 읽기/쓰기 즉시 반영
- `지연(Degraded)`: 응답 지연, 재시도 동작 중
- `오프라인(Offline)`: 서버 통신 불가, 로컬 임시 저장만 수행

## 3) UX 원칙
- 상단 고정 배너로 네트워크 상태 표시
  - 정상: 상태 점 + "정상 연결"
  - 지연: "네트워크 지연 - 재시도 중"
  - 오프라인: "오프라인 - 임시 저장 상태, 복구 시 자동 동기화"
- 저장/수정/삭제 버튼 클릭 시 즉시 피드백
  - 정상: "처리 완료"
  - 지연/오프라인: "임시 저장됨 (동기화 대기)"
- 관리자 기능(삭제/반려/권한변경)은 오프라인 시 잠금 권장

## 4) 데이터 처리 원칙
- 쓰기 작업은 모두 `작업 대기열`에 기록
  - 항목: 작업종류(create/update/delete), 대상 ID, 요청시간, 사용자 ID, 사유(삭제/반려 포함)
- 온라인 복구 시 대기열 순서대로 재실행
- 동일 예약에 대한 중복 작업은 최신 작업 1건으로 병합(coalesce)

## 5) 충돌 처리 기준
- 기본 원칙: 서버 최신 상태 우선
- 관리자 작업(삭제/반려)은 일반 수정보다 우선
- 작업자 수정이 관리자 확정 이후 들어오면 자동 반려 처리 + 사유 기록
- 충돌 발생 시 사용자에게 충돌 안내 모달 + 재시도 버튼 제공

## 6) 재시도 정책
- 백오프 간격: 2초 -> 5초 -> 10초 -> 20초 (최대 60초)
- 일정 횟수 실패 시 지연 상태 표시 + 수동 재시도 버튼 노출
- 브라우저 재시작 대비 임시 저장소(예: IndexedDB) 사용 권장

## 7) 로그/감사
- 실패 로그: 요청 유형, 대상, 사용자, 에러코드, 횟수
- 복구 로그: 대기 건수, 성공/실패 건수, 최종 처리 시각
- 관리자 화면에 "동기화 상태 요약" 영역 추가 예정

## 8) 단계별 적용 순서 (추후 코드 구현)
1. 상태 배너 + 네트워크 상태 감지
2. 쓰기 공통 래퍼 + 재시도 백오프
3. 로컬 작업 대기열 저장/복구
4. 충돌 정책 적용 + 관리자 우선 규칙
5. 운영 로그 시각화 + 리포트 반영

## 9) 테스트 시나리오
- Wi-Fi 강제 차단 후 예약 생성/수정/삭제
- 10분 오프라인 후 복구 시 순차 동기화 확인
- 동일 예약을 작업자/관리자가 각각 수정해 충돌 재현
- 모바일 환경(저속 네트워크)에서 지연 상태 메시지 확인
